# Initial Analysis

We need to implement a modular offline Sokoban game system in Java.
- **`sokoban-engine`**: Pure Java library (using Gradle's `java-library` plugin). Handles game state, direction, positions, Level XML parser, undo/redo move state, and moves/win mechanics.
- **`sokoban-android`**: Android Application project. It will depend on `:sokoban-engine`. It will implement `MainActivity`, a custom `GameView` (Canvas-based rendering), `LevelRepository` (loading XML files from `assets/levels/`), and `AndroidGameController` for input handling and UI controls (Swipe, on-screen controls, restart/undo buttons, moves counter, level selection).

# Assumptions

1. **Gradle Build Setup**: We will configure standard multi-module build via `settings.gradle` and root `build.gradle`.
2. **Java 17 / Toolchain**: The Android build uses Android Gradle Plugin 8.4.0. We will target compatibility for Java 17 for both modules.
3. **XML Parser**: We will use Java's standard XML libraries (`javax.xml.parsers.DocumentBuilderFactory` and `org.w3c.dom.*`) which are available on both standard JVM and Android platforms.

# Risks

1. **Android Canvas Rendering Performance**: Render loops must be efficient. Drawing onto a custom Canvas with `onDraw()` needs to handle screen resize, scale factor calculation, and central positioning.
2. **Invalid Level Grids**: Grid rows can have varying lengths or empty lines. The parser must normalize spaces and rows to a consistent rectangular representation.
3. **Android Lifecycle**: Standard configuration changes (like rotation) should not lose current game state. We will handle layout/orientation properly or retain the controller state.
